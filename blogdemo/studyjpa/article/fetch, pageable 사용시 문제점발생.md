유저의 쓴 글의 목록을 불러오기 위한 메소드를 작성하는데, 모든 데이터를 메모리에 올리고
페이징 처리를 해서 매우 위험한 상황이다.

이를 해결하기 위한 방법이 무엇인가에 찾아보면서 정리

# 배치 사이즈 (default_batch_fetch_size) 란?
- Batch Fetching 은 Hibernate 가 연관된 엔티티를 한 번에 가져오는 크기를 설정하는 방법이다.
- 기본적으로 FetchType.LAZY 로 설정된 연관 필드는 개별적으로 초기화되는데, N+1 문제가 발생할 가능성이 있다.
- Batch Fetching 은 이 문제를 완화하기 위해 지정된 크기만큼 데이터를 한 번에 가져오는 방식입니다.


동작 방식:
- 1_000 으로 설정한 경우
    - 한 번에 최대 1_000개의 연관된 데이터를 쿼리하여 가져옵니다.
    - @OneToMany, @ManyToOne, @OneToOne 관계에 대해 효율적으로 데이터를 로드합니다.

## 설정의 이점
1. N + 1 문제 해결:
   - 기존 방식 : 부모 엔티티를 가져온 후, 각 부모 엔터티의 연관 데이터를 개별적으로 조회 -> n+1 문제
   - 배치 페칭 : 부모 엔티티를 가져온 후, 연관 데이터를 묶어서 한 번의 쿼리로 가져옴.
2. 쿼리 성능 최적화
   - 쿼리 수를 줄여서 데이터베이스와의 상호작용을 줄이고 성능을 개선
3. 페이징과 호환 가능:
   - 페이징과 함께 사용할 때, 연관된 데이터를 효율적으로 로드

## 문제점
1. 페이징과 Fetch Join의 충돌 가능성
    - @Query에서 Join FETCH 와 pageable 을 함께 사용할 때, 메모리에서 페이징 처리할 수 있다.
    - 이는 성능 저하로 이어질 수 있으므로 DISTINCT 와 페이징이 함께 사용되는지 확인하고 테스트


## 시도
글을 조회해올 때 쿼리를 두 번으로 나누어서 설계 후 서비스에서 두 번의 호출
1. user id를 articleRepository 에 넘기면서 userid 로 article 페이지네이션해서 가져오기
2. 가져온 article 객체에서 페이징 처리 하기

- 일단 user 가 쓴 글이 매우 많거나 할 경우, 모든 데이터를 로드해서 가져오기 때문에 메모리에 많이 올라감
- 이를 해결하기 위해 batch size 를 설정할 수 있을 것 같음
- 두 번의 쿼리를 통해 가져옴

DTO 활용

article 대신 필요한 필드만 담은 DTO를 사용해 데이터 전송과 로딩 효율성을 높임

단점 : 엔터티가 아닌 DTO를 사용하는 경우 추가적인 매핑 코드가 필요할 수 있음

## Query DSL 
Query DSL을 사용하면 현재 문제를 훨씬 더 유연하고 깔끔하게 해결할 수 있다. 특히 
동적 쿼리 작성과 N+1 문제 해결에 매우 효과적이다. 

1. 동적 쿼리 작성
   - 복잡한 조건이 추가되더라도 코드 내에서 조건을 유연하게 추가/제거 가능
2. 타입 세이프
   - 쿼리를 작성할 때 컴파일 시점에 에러를 감지
3. fetch join
   - fetchJoin() 을 사용하여 연관 데이터를 함께 로딩 가능
4. 페이징 지원
   - Query DSL 을 사용하면 Pageable 을 적용해 페이징 처리와 Fetch Join을 함께 구현가능
