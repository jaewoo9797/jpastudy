## 유저가 쓴 글의 목록을 가지고오는 기능 구현
유저 `id` 로 본인이 쓴 글을 조회해오는 쿼리를 작성했습니다. 처음에는 **단방향**으로 2번의 쿼리를 디비에 보냈습니다.

쿼리 최적화를 위해 `Article` 의 객체에 `User` 와 연관관계를 맺어주고 JPQL 로 `fetch join` 과 `paging` 처리를 함께 주었는데,
모든 데이터를 데이터베이스에서 조회해 와서 메모리에 올리고 페이징 처리 하여
메모리 이슈가 생길 수 있어서 `batch size` 를 설정해주었습니다. 이것도 결국 메모리에 다올리지만 제한된 데이터를 가져오는것 같습니다.

`QueryDsl` 로 `fetch join` 과 `pageable` 을 적용해 테스트 코드를 실행해 보았는데, 이것도 해당 `User` 의 글을 모두 가져온 후 페이징
처리하는 것 같습니다.

또한 count 하기위한  두 번의 쿼리를 날리는데 이러면 최적화가 안되는것 아닌가 싶습니다., 디비의 데이터를 메모리에 다올리는 방식보다는
조회할 때 `userid` 로 `Article` 을 페이징 적용해서 `Ariticle` 의 `id`를 가져온 후
조회된 `ids` 를 통해서 article을 fetch join 하는 방식이 조금 더 낫지 않을까 싶습니다.

단점으로는 호출해야하는 메서드가 두 개라서 유지보수와 가독성이 떨어진다?

```java
    @Query("SELECT a.id FROM Article a WHERE a.user.id = :userId")
    Page<Article> findArticleIdsByUserId(@Param("userId") Long userId, Pageable pageable);

    @Query("SELECT DISTINCT a FROM Article a "
            + "LEFT JOIN FETCH a.comments "
            + "JOIN FETCH a.user "
            + "WHERE a.id IN :ids")
    List<Article> findArticlesWithDetailsByIds(@Param("ids") List<Long> ids);
```

## 정리
1. `fetch join` + `paging` 의 문제점:
    - 모든 데이터를 가져와 메모리에서 페이징 처리 -> 메모리 이슈 발생 가능.
    - `batch size` 로 완화 가능하지만, 여전히 모든 데이터를 로드해야 한다는 근본적인 문제는 남음
2. `QueryDSL` 로 `fetch join` 과 페이징 적용:
   - 동일한 메모리 페이징 문제가 발생.
   - `count` 쿼리를 위한 추가 쿼리 발생.
3. 두 단계 조회 방식:
   - `Article.id` 를 먼저 페이징 조회 -> `fetchh join`으로 필요한 데이터를 다시 조회
   - 데이터 로드를 최소화하고, 페이징 문제를 해결.
   - 단점 : 호출 메서드가 두 개로 늘어나고 가독성과 유지보수성에 불리.


### 제안 방안
두 단계 조회 방식을 채택

1) 메모리 사용 최적화 : 데이터베이스에서 페이징 처리 후, 필요한 데이터만 조회하므로 메모리 사용량 감소.
2) 쿼리 성능 최적화 : 필요한 데이터만 가져오기 때문에 데이터베이스 I/O 부담 완화
3) 페이징 문제 해결 : fetch join 의 페이징 문제를 근본적으로 회피

단점
1) 코드 복잡도 증가 : 두 개의 메서드를 호출해야 하므로 코드 가독성이 떨어질 수 있음
2) 트랜잭션 내 두 번의 쿼리 실행 : DB 호출이 두 번 발생.


Batch Fetching 병행
`fetch join` 대신 `batch size`를 사용하여 연관 데이터를 효율적으로 로드
