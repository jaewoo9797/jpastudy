Entity 는 데이터베이스에 저장되는 객체로서 해당 객체의 정보를 가지고와서 service layer
에서 setter 메서드를 통해 변경을 해준 후 변경 된 내용을 데이터베이스에 저장할 수 있다.

# 서비스 레이어에서 setter, 객체에서 수정하는 메서드
그런데, 이게 객체지향적으로 프로그래밍하는 것인가에 대해서 고민이 된다. 객체에 대한 데이터의 관
리의 책임은 해당 객체라고 생각하기 때문이다. 나에 대한 것은 자신이 가장 잘 알고 메서드를 통해
협력하여 해야지 다른 누군가 나에 정보를 강제로 바꾼다는 것은 객체지향 프로그래밍에 벗어난다고 생각
하기 때문이다.

이에 대해서 정보를 찾아보고 싶어서 gpt 에 관련 정보를 알려달라고 하고, url을 제공해달라고 했다.

## 서비스 레이어에서 엔티티를 수정하는 방식: 
일반적으로 서비스 레이어에서 엔티티를 조회한 후, 필요한 필드의 값을 변경하고, 변경된 엔티티를 
저장하는 패턴을 많이 사용합니다. 이때, JPA의 `변경 감지` 기능을 활용하여, 트랜잭션 내에서
엔티티의 필드 값을 변경하면 트랜잭션이 커밋될 때 자동으로 해당 변경 사항이 데이터베이스에 반영
됩니다. 이 방법은 명시적으로 어떤 필드가 변경되었는지 코드에서 쉽게 파악할 수 있어
유지보수에 용이합니다.

## 엔티티 내부에 메서드를 만들어 수정하는 방식:
엔티티 자체에 비즈니스 로직을 포함시켜 해당 엔티티의 상태 변경을 책임지도록 하는 방법도 있습니다.
이는 객체지향적인 설계 원칙에 부합하며, 엔티티가 자신의 상태를 스스로 관리하도록 합니다.
예를 들어, `User` 엔티티 내부에 `changeEmail(String newEmail)` 과 같은 메서드를 만들어
이 메일 변경 로직을 캡슐화할 수 있습니다. 이렇게 하면 엔티티의 일관성을 유지하고, 관련된
로직이 분산되지 않아 코드의 응집도를 높일 수 있습니다.

## 어떠한 방법이 더 객체지향적인가?
엔티티 내부에 상태 변경 메서드를 두는 것이 더 객체지향적인 접근 입니다. 이는 엔티티가 자신의
상태와 행위를 스스로 관리하도록 하여 응집도를 높이고, 코드의 재사용성과 유지보수성을 향상시
킵니다. 반면, 서비스 레이어에서 엔티티의 상태를 변경하는 것은 엔티티의 내부 구현에 대한
지식이 서비스 레이어에 노출되므로 캡슐화 원칙에 어긋날 수 있습니다.
